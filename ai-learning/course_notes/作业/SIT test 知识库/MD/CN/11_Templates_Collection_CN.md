## 如何使用本文档

本文档汇集了 SIT 项目中常用的 **6 类模板**。每个模板包含：
- **使用说明**：什么场景下使用、谁来用
- **模板正文**：可直接复制使用
- **填写示例**：基于 SIT 项目的真实场景示例

**使用方式**：复制模板正文 → 替换方括号 `[]` 中的占位内容 → 填入实际数据。

## 模板 A：SIT 测试用例模板

### 使用说明
- **使用场景**：从需求文档拆解测试用例，或为新合同类型补充测试场景
- **使用者**：甲方 PM / QA
- **输出格式**：表格（可粘贴到 Excel 或飞书多维表格）

### 模板正文

```markdown
# [功能名称] — SIT 测试用例

> 需求来源：[需求文档编号/名称]
> 测试环境：[SIT / UAT]
> 编写日期：[YYYY-MM-DD]
> 编写人：[姓名]

## 测试范围

- 涉及系统：[QT / Origination / Odessa]
- 涉及产品类型：[Finance Lease China / AOR CHINA / HKG FL]
- 前置条件：[描述测试环境要求和数据准备]

## 测试用例

| 用例编号 | 功能模块 | 测试类型 | 用例描述 | 前置条件 | 操作步骤 | 预期结果 | 优先级 | 实际结果 | 状态 |
|---------|---------|---------|---------|---------|---------|---------|--------|---------|------|
| [TC-XX-001] | [模块名] | [正常/边界/异常] | [一句话描述] | [前置条件] | 1. [步骤1] 2. [步骤2] 3. [步骤3] | [可验证的预期结果] | [P0/P1/P2] | [留空] | [留空] |

## 测试类型说明

| 类型 | 定义 | 每个功能点最少用例数 |
|------|------|-------------------|
| 正常流程（Happy Path） | 标准业务场景下的正常操作 | 2-3 条 |
| 边界条件（Boundary） | 极值、临界值、特殊值 | 2-3 条 |
| 异常场景（Negative） | 错误输入、缺失数据、违规操作 | 2-3 条 |
| 数据校验（Data Validation） | 字段格式、长度、类型校验 | 1-2 条 |
| 兼容性（Compatibility） | 跨浏览器、多语言、不同角色 | 1-2 条 |

## 测试结果汇总

- 总用例数：[X]
- 通过：[X] | 失败：[X] | 阻塞：[X] | 未执行：[X]
- 通过率：[X%]
```

## 模板 B：缺陷报告模板

### 填写示例

| 属性 | 值 |
|------|-----|
| **Bug 标题** | [Odessa-Interest Rate] 固定利率合同中 Float Rate Index 被错误填充为 LPR |
| **严重程度** | Major |
| **优先级** | P2 |
| **缺陷类型** | 集成缺陷 |
| **责任方** | 乙方 |
| **所属模块** | Odessa - Lease Interest Rate |
| **关联测试用例** | TC-OD-001 |

**重现步骤**：
1. 在 QT 中创建一个 Rate Type = Fixed 的 Direct Lease 报价
2. 通过 Origination 审批后发送至 Odessa
3. 在 Odessa Lease 模块打开该合同的 Interest Rate 标签页
4. 观察 Float Rate Index 字段

**预期结果**：根据文档 02 第 2.4 节，Rate Type = Fixed 时，Float Rate Index 应为空/禁用。

**实际结果**：Float Rate Index 显示为 "LPR"，Float Rate Lease 复选框为 Yes。

| 系统 | 字段 | 期望值 | 实际值 |
|------|------|--------|--------|
| QT | Rate Type | Fixed | Fixed |
| Origination | Rate Type | Fixed | Fixed |
| Odessa | Float Rate Index | 空/禁用 | LPR |
| Odessa | Float Rate Lease | No | Yes |

## 模板 E：AI 辅助提示词模板

### 使用说明
- **使用场景**：利用 AI（如 Claude）辅助生成测试用例、Bug 报告、验收清单
- **使用者**：甲方 PM
- **使用方式**：复制提示词 → 替换方括号内容 → 粘贴给 AI

### E1: 测试用例生成提示词

```
你是一位资深QA工程师，熟悉多系统集成测试（SIT）。请根据以下需求信息，生成完整的SIT测试用例。

## 需求信息
- 功能名称：【填写功能名称】
- 需求描述：【粘贴需求文档或用户故事】
- 涉及系统：【QT / Origination / Odessa】
- 涉及产品类型：【Finance Lease China / AOR CHINA / HKG FL】
- 前置条件：【填写测试环境和数据要求】

## 输出要求
请按以下表格格式输出测试用例：

| 用例编号 | 功能模块 | 测试类型 | 用例描述 | 前置条件 | 操作步骤 | 预期结果 | 优先级 |

要求：
1. 每个功能点至少覆盖：正常流程 2-3 条、边界条件 2-3 条、异常场景 2-3 条
2. 操作步骤要具体到可执行，不能模糊
3. 预期结果要可验证，不能写"正常显示"这种模糊描述
4. 优先级：P0（必测）、P1（重要）、P2（建议）
5. 特别关注：三系统间数据流转完整性、中国/香港差异、利率类型条件逻辑、还款方式影响
```

### E2: 缺陷报告生成提示词

```
你是一位资深QA工程师。请将我发现的SIT测试问题整理成标准Bug报告。

## 我发现的问题
【用口语描述即可，例如："在Odessa的Interest Rate标签页，明明QT选的是Fixed利率，但Float Rate Index显示了LPR"】

## 相关背景
- 所属模块：【QT / Origination / Odessa - 具体标签页】
- 测试环境：SIT
- 合同类型：【如 TC1 Vendor Program DL】
- 产品类型：【Finance Lease China / AOR / HKG FL】

## 参考知识
- 利率类型规则见文档 02 第 2.4 节
- 系统默认值见文档 02 第 2.2 节
- 缺陷分类决策树见文档 08 第 8.1 节

## 输出要求
请按以下格式输出：
1. Bug标题（[模块] + 简洁问题描述）
2. 严重程度（Critical/Major/Minor/Trivial）和优先级（P1-P4）
3. 缺陷类型（集成/Origination/配置/计算/测试数据）和责任方（甲方/乙方）
4. 重现步骤（具体到每一步操作）
5. 预期结果（引用知识库文档依据）
6. 实际结果
7. 数据追溯表（QT → Origination → Odessa 三系统字段值对比）
8. 影响范围和附件建议
```

### E3: 缺陷模式分析提示词

```
你是一位资深QA分析师。请根据以下缺陷数据，分析缺陷模式并生成改进建议。

## 缺陷数据
【粘贴缺陷列表，包含：缺陷编号、模块、类型、优先级、状态】

## 分析要求
1. 按模块统计缺陷分布
2. 按类型（集成/配置/计算/数据）分析占比
3. 识别高频缺陷模式（是否存在系统性问题）
4. 给出根因假设和修复建议
5. 评估对未执行测试用例的潜在影响

## 输出格式
- 缺陷分布图表（文字表格）
- Top 3 缺陷模式及根因分析
- 给乙方的修复优先级建议
```
