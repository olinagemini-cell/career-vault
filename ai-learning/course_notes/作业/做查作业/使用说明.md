# 需求分析「做 + 查」配套工具使用说明

## 工具1：Skills 技能包（执行步骤）

| 项目 | 内容 |
|------|------|
| **技能名称** | `req-analysis` |
| **文件位置** | `~/.claude/skills/req-analysis/SKILL.md` |
| **调用方式** | 输入 `/req-analysis` 或说"做需求分析"、"业务需求转IT方案"、"需求梳理" |
| **用途** | 按标准10步流程，将业务需求转化为结构化的IT需求规格方案 |
| **适用场景** | 跨国IT系统在本地落地（融资租赁/金融业务），总部系统能力 → 本地业务需求 → IT规格 |
| **何时使用** | 接收业务需求后、开始写需求文档前 |

### 使用方式

1. 触发 `/req-analysis`
2. 提供三个背景信息：业务场景、系统背景、已有输入
3. 跟着10步流程走：系统调研 → 他国PM → 本地调研 → 反向确认 → 战略对齐+模块验证 → 分级+伪需求+ROI → 冲突处理 → 总部拉齐 → 交付 → 追踪
4. 产出：符合9段标准结构的需求规格文档

### 示例对话

**用户**：`/req-analysis`

**Claude**：我会帮你按10步流程完成需求分析。先确认三个背景信息：
1. **业务背景**：什么业务场景（销售/风控/合同管理/...）？涉及哪些部门？
2. **系统背景**：总部系统是什么？本地上线的阶段（全新落地/迭代优化）？
3. **已有输入**：是否已看过Demo？是否有他国PM的反馈？是否有现存资料？

**用户**：业务背景是中国区融资租赁报价流程，涉及销售部门和风控审批。系统背景是总部QT Application全新落地中国，Demo已看过，也跟日本和新加坡的PM聊过。已有输入是销售反馈中国必须含税报价，但总部系统只支持net报价。

**Claude**：清楚了。我们按流程走：
- Step 1 先梳理QT的系统架构和报价引擎逻辑...
- Step 2 回顾日本/新加坡PM的反馈，看有没有类似的含税需求...
- [逐步骤推进]

## 工具2：Commands 斜杠命令（质量检查）

| 项目 | 内容 |
|------|------|
| **命令名称** | `check-req-spec` |
| **文件位置** | `~/.claude/commands/check-req-spec.md` |
| **调用方式** | 输入 `/check-req-spec` 或说"检查需求文档"、"需求质量检查" |
| **用途** | 从8个维度检查需求规格文档的质量，产出评分+改进建议 |
| **何时使用** | 需求规格文档完成后，提交给总部/开发/评审前 |

### 检查的8个维度

| # | 维度 | 核心问题 |
|---|------|---------|
| 1 | 需求清晰度 | 开发和测试能直接看懂吗？ |
| 2 | 计算逻辑完整性 | 所有公式变量列全了吗？场景变体覆盖了吗？ |
| 3 | 可追溯性 | 每条需求能找到业务来源吗？ |
| 4 | 业务冲突显性化 | 矛盾点被记录了吗？有决策人和时间线吗？ |
| 5 | 伪需求过滤 | Blocker的分级依据能复核吗？ |
| 6 | 验收标准可测 | 正常+异常场景都能验证吗？ |
| 7 | 总部沟通就绪度 | 总部能独立看懂并进入方案讨论吗？ |
| 8 | 战略与模块对齐 | 每条需求有战略归属和模块归属吗？范围蔓延被控制了吗？ |

## 配套使用流程

```
1. 用 /req-analysis（Skill）按9步流程完成需求分析
       ↓
2. 得到需求规格文档
       ↓
3. 用 /check-req-spec（Command）从7维度质量检查
       ↓
4. 根据检查报告修改文档
       ↓
5. 再次检查，直到全部通过
       ↓
6. 提交给总部/开发/评审
```

## 使用建议

- **每次需求分析都用这套**：先 `/req-analysis` 走流程，再 `/check-req-spec` 查质量
- **第一次用的时候多花时间**：Step 5（分级+伪需求）和 Step 7（总部拉齐）是最体现PM专业度的环节
- **计算对照法高度推荐**：金融系统的计算逻辑冲突，让业务方手算 vs 系统逻辑直接对比，PM不要传话
- **定期回顾**：上线后的使用数据（Step 9）反哺下一次的需求判断
