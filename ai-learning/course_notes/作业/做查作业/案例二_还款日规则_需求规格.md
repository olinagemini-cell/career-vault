# 需求规格：中国区固定还款日规则——多系统一致性方案

---

## 1. 执行摘要（30秒读完）

> **问题**：中国区还款日固定为每月10号和20号（非每日），且50%客户需要不规则还款。这涉及QT（报价）、ORIG（审批）、Odessa（合同）三个系统的对账——三套引擎必须输出一致结果。
>
> **方案选项**：ORIG作为唯一计算源，QT/Odessa只读取不重算。自研系统（QT/ORIG）总部改，外采系统（Odessa）单独评估改造边界。
>
> **需要总部做什么**：确认ORIG承担中国区计算引擎角色；评估Odessa是否支持自定义还款计划，不支持的话中国区做兼容层还是Odessa厂商二次开发。

---

## 2. 需求描述 + 业务场景

### 业务背景

中国区融资租赁业务涉及销售→风控→运营→财务→催收全链条。总部系统预设"放款日=还款锚点日，每日均可还款"，但中国区为降低管理成本，还款日固定为每月10号和20号。

### 业务规则

| 放款日区间 | 先付 | 后付 |
|-----------|------|------|
| 1-15号 | 当月10号 | 下一周期10号（月度=次月10，季度=下季首月10） |
| 16-31号 | 当月20号 | 下一周期20号（同上） |
| 不规则 | 客户自定义（占50%） | 客户自定义（占50%） |

### 跨部门影响链

```
销售报价（先付/后付/不规则已知，但放款日未知）
  → ORIG审批（确定实际放款日）
    → 还款日映射（10/20规则 or 自定义）
      → IRR重算（现金流时点变了）
        → margin重算（= IRR - COF）
      → 还款计划表生成 → Odessa合同/还款文件
      → 逾期利息起算日 = 还款日+1 → 催收
```

### 涉及部门

| 部门 | 影响 |
|------|------|
| 销售 | 报价时放款日未知，需双场景预估IRR区间 |
| 风控 | 审批后确定放款日→触发还款日映射→IRR重算 |
| 财务 | IRR/margin依赖还款日，租金不变但IRR变了 |
| 运营 | 还款文件、合同按10/20/自定义生成 |
| 催收 | 逾期利息以实际还款日为起算基准 |

## 3. 系统影响矩阵

### 系统全景图

| 层级 | 系统 | 来源 | 角色 | 是否独立计算 | 可改范围 | 单点风险 |
|------|------|------|------|------------|---------|---------|
| 用户端 | QT | 自研 | 报价引擎 | ❌ 读取ORIG的IRR回显，不做独立计算 | 中（自研，但改QT影响他国报价界面） | 如果QT自行重算，IRR跟ORIG不一致 |
| 用户端 | ORIG | 自研 | 审批+**单一计算源** | ✅ **唯一IRR/还款日计算源** | 高（自研，但改造需评估对他国审批流程影响） | 如果ORIG计算逻辑没改对，下游全错 |
| 用户端 | Odessa | 外采 | 合同管理 | ❌ 读取ORIG结果，不做独立计算 | 低（外采，每个改动=厂商评估+商务谈判+排期） | 如果Odessa不支持自定义还款计划，合同无法自动生成 |
| 控制层 | 工作流引擎 | 自研 | 审批流程+状态机 | — | 低（工作流配置层，改审批节点和触发规则） | 如果还款日确认节点没加入工作流，流程断档 |
| 数据层 | QT/ORIG/Odessa 各自DB | 自研+外采 | 数据持久化 | — | 中（自研DB可改schema，Odessa DB不可控） | 字段mapping不对齐→同一变量三套值 |
| 外部服务 | KYC | 外采 | 客户身份校验 | — | 不可改 | 不直接影响还款计算，但客户数据映射需一致 |

### 单一计算源指定

| 变量 | 计算源 | 其他系统 | 同步机制 |
|------|--------|---------|---------|
| **还款日** | ORIG（放款后唯一确定） | QT读取回显，Odessa读取写入合同 | ORIG→API→QT/Odessa |
| **IRR** | ORIG（基于实际还款日+现金流） | QT读取回显，Odessa读取写入合同 | ORIG→API→QT/Odessa |
| **margin** | ORIG（= IRR - COF） | QT读取回显，Odessa不存储 | ORIG→API→QT |
| **还款计划表** | ORIG生成 | Odessa读取后写入合同模板 | ORIG→API→Odessa |
| **逾期利息** | Odessa（基于还款日+1） | 催收系统读取 | Odessa→催收系统 |

### Odessa 改造边界评估

| 评估项 | 状态 |
|--------|------|
| 是否支持读取外部还款日（非自行计算）？ | **待确认**——当前Odessa可能内置了"放款日=还款日"逻辑 |
| 是否支持自定义还款计划录入？ | **待确认**——50%中国客户需要不规则还款 |
| 合同模板是否可配置还款日变量？ | **待确认**——目前模板可能硬编码了还款日字段 |
| 合同范围覆盖多少改造量？ | **待确认**——需拉Odessa厂商合同附件确认 |
| 厂商响应周期预估 | **待确认**——外采通常4-8周评估+开发 |

## 4. 计算逻辑规格

### 变量清单

| 变量 | 取值来源 | 取值时点 | 备注 |
|------|---------|---------|------|
| 放款日 | ORIG审批完成后确定 | 审批通过日 | 销售报价时未知 |
| 还款日 | = f(放款日, 先付/后付/不规则, 10/20规则) | 放款日确定后即时计算 | 单一计算源=ORIG |
| 先付/后付标志 | 销售与客户协商，报价时输入 | 报价创建时 | 已知 |
| 不规则还款计划 | 客户自定义，销售录入 | 报价创建时 | 占50%客户 |
| 租金 | 设备金额/租期计算 | 报价创建时 | **不变** |
| IRR | XIRR(含还款日+SD+管理费的所有现金流, 各现金流实际日期) | 还款日确定后重算 | 单一计算源=ORIG |
| COF | 总部资金成本配置 | 报价时 vs 放款时可能不同 | 待总部确认以哪个时点为准 |
| margin | = IRR - COF | 还款日确定后重算 | 单一计算源=ORIG |
| 税率 | 中国增值税率 | 报价创建时 | — |
| SD | 合同约定（≈1-2月租金） | 报价创建时 | 100%订单涉及 |

### 10/20 映射函数

```
def 还款日(放款日, 付款方式, 周期):
    if 不规则:
        return 客户自定义还款计划

    if 放款日 between 1 and 15:
        基准日 = 当月10号 if 先付 else 下一周期10号
    else:  # 16-31
        基准日 = 当月20号 if 先付 else 下一周期20号

    return 基准日
```

### 场景变体覆盖

| 场景 | 放款日 | 先付/后付 | 周期 | 预期还款日 |
|------|--------|----------|------|-----------|
| A | 3月8日 | 先付 | 月度 | 3月10日 |
| B | 3月8日 | 后付 | 月度 | 4月10日 |
| C | 3月18日 | 先付 | 月度 | 3月20日 |
| D | 3月18日 | 后付 | 月度 | 4月20日 |
| E | 3月18日 | 后付 | 季度 | 7月20日（下一季度首月=4月→7月? 需总部确认季度周期起算规则） |
| F | 任意 | 不规则 | — | 客户自定义计划 |
